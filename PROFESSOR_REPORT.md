> 報告對象：指導教授　撰寫日期：2026-07-23

## 一、為什麼從純 Python 套件改成現在的混合式架構

### 1.1 原始做法

系統最初是單純的 Python 後端（FastAPI + `pandas` / `numpy` / `statsmodels` / `pingouin` / `factor_analyzer`），所有 PLS-SEM 相關計算都用 Python 統計套件自己實作：信度用 `pingouin.cronbach_alpha`、因素負荷量用 `factor_analyzer` 的單因子萃取、路徑顯著性用自己寫的 bootstrap 迴圈。

### 1.2 這個做法的限制

PLS-SEM 這個方法論領域，業界公認的標準參照工具是 SmartPLS（商用軟體）。學術界要驗證一套自行開發的統計實作「算得對不對」，慣例做法是拿同一份資料去跟 SmartPLS 或跟業界公認正確的開源實作對答案。

Python 生態圈裡雖然有零星的 PLS-SEM 相關套件，但沒有一個被廣泛驗證、跟 SmartPLS 結果對得上的成熟實作；尤其是完整的多因子探索性因素分析（EFA）搭配 Parallel Analysis 判斷因子保留數量，Python 端幾乎沒有跟 R 的 `psych` 套件同等成熟度的替代品。這代表如果全部用 Python 自己刻，這套系統算出來的每一個數字，都缺乏一個可信賴的外部對照基準——對一個要拿去支撐論文結論的分析工具而言，這是不能接受的風險。

### 1.3 為什麼選擇 R

R 的 `seminr` 套件是由 SmartPLS 原始開發團隊（Ringle, Wende, Becker 等人）親自撰寫的開源實作，設計目標就是在 R 裡複刻 SmartPLS 的演算法，結果與 SmartPLS 高度一致，也被大量 PLS-SEM 方法論論文引用。`psych` 套件則是 EFA + Parallel Analysis 這個領域公認最成熟、最常被方法論文獻引用的實作。用這兩個套件當統計核心，系統算出來的每一項指標都有明確的、被學界廣泛使用的參照依據，不是自己閉門造車的黑箱數字。

### 1.4 為什麼不是純 R

但系統的完整願景不只是「算統計」，還包含讓使用者能上傳問卷、看儀表板、之後還要能針對分析結果互動提問（LLM 對話功能）。這一塊涉及網頁服務、非同步請求處理、對話狀態管理——R 的網頁服務生態圈（如 `plumber`）遠不如 Python 的 FastAPI 成熟，尤其是要接 LLM API、做串流回應這類現代網頁應用的需求。

### 1.5 最終決策：混合式架構

因此採用**混合式架構**：Python/FastAPI 作為對外的主服務，負責檔案上傳、使用者互動、整體流程控制；R 作為內部的統計運算核心，只負責「拿資料進來、把統計結果算出來」，兩者之間用標準的程序呼叫串接。這個分工讓系統同時取得「統計結果有業界標準工具背書」跟「網頁服務工程成熟度」兩邊的好處，而不用二選一犧牲其中一項。

## 二、現在系統的完整架構說明

### 2.1 整體架構

```
使用者（瀏覽器）
     │
     ▼
前端（兩種並存）：Gradio 對話式介面（單一聊天串完成上傳／宣告／診斷／討論／重跑）＋ Streamlit 分步驟頁面（L0-L5 各自獨立頁面，適合逐步展示方法論或需要手動細節控制的情境）
     │  HTTP API 呼叫
     ▼
Python / FastAPI 主服務
  - 檔案上傳與解析
  - 流程控制、使用者 session 管理
  - 統計計算：內建 Python 近似引擎（預設） / 呼叫 R 執行完整 PLS-SEM（選用）
  - 寫入審計資料庫
     │  程序呼叫（僅選用 R 引擎時才會走這條，呼叫一次、執行完、拿到結果）
     ▼
R 統計運算核心（seminr + psych，選用）
  - 完整多因子 EFA + Parallel Analysis
  - 完整 PLS-SEM：信效度、HTMT、路徑顯著性、R²、f²、Q²predict
     │
     ▼
SQLite 資料庫
  - 研究宣告（含時間戳記）
  - 上傳的每一版資料
  - 每一次分析/優化操作的完整、不可竄改紀錄
```

（架構圖裡特別標出「預設」跟「選用」，是因為系統實際上有**兩套平行的統計計算引擎**，不是只有 R 這一套——細節見 2.4。）

### 2.2 設計原則：三篇方法論文獻決定了架構的形狀

這套系統的架構不是憑空設計的，三個核心設計決策各自對應一篇方法論文獻：

**原則一：測量模型與結構模型必須分兩階段評估，且前者是後者的硬性前提**

> Hair, J. F., Hult, G. T. M., Ringle, C. M., & Sarstedt, M. *A Primer on Partial Least Squares Structural Equation Modeling (PLS-SEM)*. Sage Publications.

PLS-SEM 的標準流程是先確認測量模型的信效度過關，才進行結構模型的路徑顯著性檢定。系統把這件事做成程式碼層面的**強制關卡**：如果測量模型（信度、AVE）沒有通過門檻，系統會直接拒絕執行結構模型分析（回傳 403 錯誤），除非使用者明確登記理由強制略過，而這個略過動作會被完整記錄下來。

**原則二：排除任何一筆樣本都需要「統計 + 實質」雙重理由，不能只憑一個統計數字**

> Curran, P. G. (2016). Methods for the detection of carelessly invalid responses in survey data.

單一指標（例如只看 Cook's Distance 這類統計影響力指標）並不足以構成「這筆資料有問題該排除」的理由。系統實作了多訊號收斂偵測（Mahalanobis 距離、組內作答變異量 IRV、連續重複作答 Long-string），**必須至少兩個獨立訊號同時成立**，才會把某筆樣本標記為「建議複查」；而系統的自動優化引擎在考慮要不要排除某筆樣本以換取路徑顯著性時，**同時要求這筆樣本統計上有影響力、又符合實質理由標記，兩個條件缺一不可**才會執行排除。

**原則三：驗證性與探索性分析必須明確標示，否則等同於未揭露的研究者自由度**

> Simmons, J. P., Nelson, L. D., & Simonsohn, U. (2011). False-Positive Psychology: Undisclosed Flexibility in Data Collection and Analysis Allows Presenting Anything as Significant. *Psychological Science*.

Simmons 等人證明，未揭露的分析彈性（例如可以自由決定要不要多刪幾筆資料）可以把名目 5% 的偽陽性率推高到 60%。系統要求研究者在跑任何分析之前，先透過「宣告」機制把理論假設（構面、題項、假設路徑）記錄下來並蓋上時間戳記，作為驗證性分析的基準點；任何在此之後執行的「搜尋顯著性」動作，都在系統的操作紀錄裡被明確標記為探索性（exploratory），跟原始宣告的驗證性假設清楚分開。

### 2.3 系統分層架構（L0 至 L6）

| 層級 | 名稱 | 功能 |
|---|---|---|
| L0 | 研究設計宣告層 | 上傳資料前，先宣告構面、題項、假設路徑，作為驗證性/探索性分析的分界時間點 |
| L1 | 資料品質層 | 多訊號收斂偵測可能隨意作答的樣本，只診斷不刪除 |
| L2 | 測量模型層 | 信度、收斂效度、區辨效度檢定，是否過關決定能不能繼續往下分析 |
| L3 | 結構模型層 | 路徑顯著性、解釋力、預測力等結構模型指標 |
| L4 | 自動優化引擎 | 分兩階段自動搜尋：先確保測量模型過關，再針對不顯著的路徑嘗試找出最小幅度的合理調整 |
| L5 | 審計與版本層 | 每一步操作的完整、不可竄改紀錄，可完整重放從原始資料到最終數字的每一步 |
| L6 | 呈現層 | 使用者操作介面 |

### 2.4 雙引擎設計：Python 近似版 vs. R/seminr 完整版

系統實際上有**兩套平行的統計計算引擎**，這是理解所有數字從哪裡來的關鍵前提：

| | **Python 內建近似引擎**（`app/stats_engine.py`） | **R/seminr 完整引擎**（`r/seminr_wrapper.R`） |
|---|---|---|
| 觸發端點 | `/analyze/measurement`、`/analyze/structural`、`/optimize/*`（**L4 優化引擎全部走這條**）、對話介面的 `run_full_pipeline`／`rerun_optimization` 工具 | `/analyze/seminr`（Streamlit 頁面「用 R/seminr 算完整版」勾選框、`/analyze/composite` 選 `weighting=pls` 時） |
| 潛在變數分數 | 構面內題項的**簡單平均**（unweighted mean） | 真正的 **PLS 迭代演算法權重**（`estimate_pls()`，反覆疊代到收斂） |
| Loadings / AVE / CR | 單因子 EFA（`factor_analyzer` 套件，`FactorAnalyzer(n_factors=1)`）估出的近似負荷量 | seminr 內建的正式 PLS 公式 |
| 路徑顯著性 | 對標準化後的潛在變數分數跑 OLS，**手動寫的 bootstrap 迴圈**（見 2.5） | seminr 內建的 `bootstrap_model()` |
| 額外指標 | 無 HTMT／f²／Q²predict | 有 HTMT（區辨效度）、f²（效果量）、Q²predict（樣本外預測力，Shmueli et al. 2019） |
| 為什麼要分兩套 | **速度**：L4 優化引擎的 Stage B 要對同一條路徑反覆刪樣本、重跑迴歸上百次（見 2.6），如果每次都要等真正 PLS 迭代收斂，搜尋會慢到不可用 | **嚴謹**：正式要寫進論文的數字，應該用業界標準演算法算出來的版本，不是近似值 |

**這個設計對使用結果的實務意義**：L4 優化引擎（包括對話介面裡的自動分析與重跑功能）給出的「哪條路徑顯著、AVE 多少」等數字，都是 Python 近似引擎算出來的，可以拿來快速反覆試探、找方向；但**最終要寫進論文的數字，建議用同一組構面/路徑設定，額外跑一次 `/analyze/seminr`（或 Streamlit 頁面的完整版勾選框）用真正 PLS 演算法重新確認一次**，兩者不會逐位數字完全相同（近似 vs. 迭代收斂的演算法本來就不同），但方向性結論（哪些構面過關、哪些路徑顯著）在資料品質正常的情況下應該一致；如果兩邊結論不一致，代表資料可能落在近似值誤差較大的邊界情況，值得進一步檢查。

### 2.5 統計計算細節（逐層公式）

**L1 資料品質層（`detect_careless_responses`）—— 四個獨立訊號，多信號收斂偵測：**

| 訊號 | 計算方式 | 判定門檻 |
|---|---|---|
| Mahalanobis 距離 | 每位受訪者所有題項答案視為一個向量，計算與全體平均答案模式的多變量馬氏距離 | 用卡方分布反算門檻，`alpha = 0.001`（distribution df = 題項數） |
| IRV（組內作答變異量） | 同一受訪者所有題項答案的列標準差（row-wise SD），太低代表整份答案幾乎沒有變化（灌水／隨便勾同一格） | 全體受訪者 IRV 分布的最低 5% 分位數（`irv_percentile = 5.0`）以下 |
| Long-string | 同一受訪者連續填答相同選項的最長連續段長度 | ≥ 題項數的 50%（`long_string_ratio = 0.5`，至少 2 題起跳） |
| 作答時間（選填） | 若資料含作答時間欄位，取完成最快的受訪者 | 最快 5% 分位數（`fast_response_percentile = 5.0`）以下 |

**判定規則**（對應 Curran 2016 的方法論建議）：單一訊號觸發不算數，必須**至少 2 個獨立訊號同時成立**（`min_signals = 2`，可調整）才會把某筆樣本標記為「建議複查」。這一層純粹診斷、標記，**不會刪除任何資料**；只有 L4 優化引擎在 Stage B 決定要不要排除某筆樣本時，才會參照這裡的標記結果（見 2.6）。

**L2 測量模型層：**

| 指標 | Python 近似引擎算法 | R/seminr 完整引擎算法 | 門檻 |
|---|---|---|---|
| Cronbach's α | `pingouin.cronbach_alpha()` | seminr `summary()` 內建 | ≥ 0.7 |
| Loadings | 單因子 EFA 萃取（`factor_analyzer`），符號依平均負荷量方向校正為正 | PLS 迭代收斂的 outer weights/loadings | 個別題項 < 0.7 會被標記「低負荷量」 |
| AVE（收斂效度） | 各題 loading 平方的平均值：`mean(loading²)` | seminr 內建正式公式 | ≥ 0.5 |
| CR（組合信度） | `(Σloading)² / [(Σloading)² + Σ(1-loading²)]` | seminr 內建正式公式（rhoC） | ≥ 0.7 |
| Cross-loadings（區辨效度粗略檢查） | 每題與「自己構面」simple-mean 潛在分數的相關性，須比與「其他任一構面」的相關性高至少 0.10，否則標記警告/建議刪除 | 無對應獨立指標（R 引擎改用更嚴謹的 HTMT） | gap ≥ 0.10 |
| 反向題偵測 | 題項答案與自己構面平均分的相關性為負，標記建議反向編碼 | 無對應獨立指標 | 相關係數 < 0 |
| HTMT | 無 | R/seminr 內建，比傳統 Fornell-Larcker 準則更敏感 | 嚴格 0.85 / 寬鬆 0.90（Henseler, Ringle & Sarstedt, 2015） |
| 多因子 EFA + Parallel Analysis | 無（Python 引擎只做單因子驗證性 EFA，不做探索性因子保留數判斷） | `/analyze/efa` 端點，走 R `psych` 套件 | 因子保留數由 Parallel Analysis 決定，非人工主觀判斷 |

**L2 硬性關卡**：任一構面的 AVE 無法透過 L4 Stage A 的貪婪刪題救到達標，`/analyze/structural`（以及依賴它的下游端點）會直接回傳 403，阻止結構模型分析繼續，除非明確帶 `override_l2_gate=true` 並附上理由（會寫進不可竄改的審計紀錄）。

**L3 結構模型層：**

| 指標 | Python 近似引擎算法 | R/seminr 完整引擎算法 |
|---|---|---|
| 路徑係數 + 顯著性 | 潛在變數（簡單平均）標準化後跑 OLS 拿原始 β；**手動 bootstrap**：重複抽樣（預設 300-500 次）重跑迴歸，取 bootstrap 分佈標準差算 t 值（`t = |原始β / bootstrap 標準差|`），p 值以常態分布近似計算；顯著門檻 p < 0.05 且 t > 1.96 | seminr 內建 `bootstrap_model()`，真正的 PLS bootstrap 程序 |
| VIF | `statsmodels` 的 `variance_inflation_factor()`，對迴歸自變數逐一計算 | seminr `summary()` 內建 |
| R² | OLS 迴歸的判定係數，依 Cohen 分級（≥0.75 強／≥0.5 中／≥0.25 弱／其餘極弱） | seminr `summary()` 內建，另有調整後 R² |
| f² | 無 | seminr 內建，個別前因變數對解釋力的效果量貢獻 |
| Q²predict（PLSpredict） | 無 | seminr `predict_pls()`，10-fold 交叉驗證，取代傳統但爭議較大的 blindfolding 方法（Shmueli et al., 2019） |

需特別說明：**SRMR（標準化殘差均方根，整體配適度指標）目前刻意未實作**，兩套引擎都沒有。PLS-SEM 並不像傳統共變異數導向 SEM（CB-SEM）有一套幾乎強制報告配適度指標的傳統，SRMR 是後來才被借用過來的近似指標，其門檻在 PLS-SEM 情境下並未被嚴謹驗證，連 Hair 等人教科書都提醒使用時需謹慎。經評估，本系統已具備的信效度與解釋/預測力指標已涵蓋 PLS-SEM 論文的必要報告項目，SRMR 非必要項目，貿然自行實作一個沒有驗證管道的公式，風險大於效益。

### 2.6 自動優化引擎：分兩階段搜尋演算法

這是系統的核心功能，用來處理「測量模型或結構模型未達標時，該如何以最小幅度的合理調整讓分析結果站得住腳」這個問題，取代傳統上研究者手動反覆嘗試刪題項/刪樣本的過程。**整個優化引擎（Stage A + Stage B）都跑在 2.4 提到的 Python 近似引擎上**，理由同樣是速度——Stage B 對單一條路徑就可能要反覆重跑上百次迴歸，換成真正 PLS 迭代演算法會慢到不可用。

**Stage A（測量模型關卡）**：對每個未通過信效度標準的構面，採用貪婪演算法，優先刪除因素負荷量最低的題項，直到 AVE 達標或剩兩題仍未達標為止（後者會回報「無法自動修復，建議研究者重新檢視此構面」，不會勉強繼續）。這個階段是**強制關卡**——沒有全數通過，後續的結構模型搜尋完全不會執行。

**Stage B（結構顯著性搜尋，僅在 Stage A 全部通過後才啟動）**：針對每一條尚未達到統計顯著的假設路徑，各自獨立搜尋。搜尋方式是計算每筆樣本對該路徑迴歸結果的統計影響力（Cook's Distance），依影響力大小排序，逐步嘗試排除；**但每一筆候選排除的樣本，都必須同時是 L1 資料品質層標記過的「疑似隨意作答」樣本，兩個條件缺一不可才會真的排除**，且刪除比例設有上限（預設 10%）。若在此限制內仍找不到讓路徑顯著的排除組合，系統會如實回報「無法在有實質理由的前提下找到解法」，並附上給研究者參考的檢討建議（例如：是否有中介變數、是否需要重新檢視理論架構），**但不會自動刪除構面或改變理論模型**。

這個雙重限制（統計影響力 + 實質理由）是整套系統方法論嚴謹度的核心：實測驗證過，同一組資料，如果拿掉「必須有實質理由」這個限制，統計搜尋確實能找到讓某條沒有真實效果的路徑「看起來顯著」的刪除方式；加回這個限制後，系統正確地拒絕了這個搜尋，只留下建議研究者人工判斷——避免系統淪為自動化的資料操弄（p-hacking）工具。

**實務建議**：Stage A/B 找出的「最終應該用哪個構面分組、刪哪些樣本」這組決定，是這個系統對研究者最有價值的產出；但既然搜尋過程本身用的是近似引擎，一旦決定了最終的構面/樣本設定，應該用 R/seminr 完整引擎針對這組**已經定案**的設定重新算一次正式數字（含 HTMT、f²、Q²predict），才是要寫進論文的版本——優化引擎負責「探路」，R/seminr 負責「定案」。

### 2.7 部署環境

系統以 Docker Compose 管理三個容器：Python/FastAPI 主服務（內含 R 執行環境）、Gradio 對話式前端、Streamlit 分步驟前端，兩個前端各自獨立部署（不同 port），共用同一個後端與 session 機制。統計資料庫採用 SQLite，優先考量是部署簡便、零額外設定成本，符合目前開發階段的規模需求，未來若有正式上線的量能需求，可平順遷移至 PostgreSQL 等正式資料庫，不需重寫應用邏輯。
