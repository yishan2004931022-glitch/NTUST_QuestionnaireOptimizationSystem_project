## 一、為什麼從純 Python 套件改成現在的混合式架構

### 1.1 原始做法

系統最初是單純的 Python 後端（FastAPI + `pandas` / `numpy` / `statsmodels` / `pingouin` / `factor_analyzer`），所有 PLS-SEM 相關計算都用 Python 統計套件自己實作：信度用 `pingouin.cronbach_alpha`、因素負荷量用 `factor_analyzer` 的單因子萃取、路徑顯著性用自己寫的 bootstrap 迴圈。

### 1.2 這個做法的限制

PLS-SEM 這個方法論領域，業界公認的標準參照工具是 SmartPLS（商用軟體）。學術界要驗證一套自行開發的統計實作「算得對不對」，慣例做法是拿同一份資料去跟 SmartPLS 或跟業界公認正確的開源實作對答案。

Python 生態圈裡雖然有零星的 PLS-SEM 相關套件，但沒有一個被廣泛驗證、跟 SmartPLS 結果對得上的成熟實作；尤其是完整的多因子探索性因素分析（EFA）搭配 Parallel Analysis 判斷因子保留數量，Python 端幾乎沒有跟 R 的 `psych` 套件同等成熟度的替代品。這代表如果全部用 Python 自己刻，這套系統算出來的每一個數字，都缺乏一個可信賴的外部對照基準——對一個要拿去支撐論文結論的分析工具而言，這是不能接受的風險。

### 1.3 為什麼選擇 R

R 的 `seminr` 套件是由 SmartPLS 原始開發團隊（Ringle, Wende, Becker 等人）親自撰寫的開源實作，設計目標就是在 R 裡複刻 SmartPLS 的演算法，結果與 SmartPLS 高度一致，也被大量 PLS-SEM 方法論論文引用。`psych` 套件則是 EFA + Parallel Analysis 這個領域公認最成熟、最常被方法論文獻引用的實作。用這兩個套件當統計核心，系統算出來的每一項指標都有明確的、被學界廣泛使用的參照依據，不是自己閉門造車的黑箱數字。

### 1.4 為什麼不是純 R

但系統的完整願景不只是「算統計」，還包含讓使用者能上傳問卷、看儀表板、之後還要能針對分析結果互動提問（LLM 對話功能）。這一塊涉及網頁服務、非同步請求處理、對話狀態管理——R 的網頁服務生態圈（如 `plumber`）遠不如 Python 的 FastAPI 成熟，尤其是要接 LLM API、做串流回應這類現代網頁應用的需求。

### 1.5 最終決策：混合式架構

因此採用**混合式架構**：Python/FastAPI 作為對外的主服務，負責檔案上傳、使用者互動、整體流程控制；R 作為內部的統計運算核心，只負責「拿資料進來、把統計結果算出來」，兩者之間用標準的程序呼叫串接。這個分工讓系統同時取得「統計結果有業界標準工具背書」跟「網頁服務工程成熟度」兩邊的好處，而不用二選一犧牲其中一項。

## 二、現在系統的完整架構說明

### 2.1 整體架構

```
使用者（瀏覽器）
     │
     ▼
前端（兩種並存）：Gradio 對話式介面（單一聊天串完成上傳／宣告／診斷／討論／重跑）＋ Streamlit 分步驟頁面（L0-L5 各自獨立頁面，適合逐步展示方法論或需要手動細節控制的情境）
     │  HTTP API 呼叫
     ▼
Python / FastAPI 主服務
  - 檔案上傳與解析
  - 流程控制、使用者 session 管理
  - 呼叫 R 執行統計運算
  - 寫入審計資料庫
     │  程序呼叫（呼叫一次、執行完、拿到結果）
     ▼
R 統計運算核心（seminr + psych）
  - 完整多因子 EFA + Parallel Analysis
  - 完整 PLS-SEM：信效度、HTMT、路徑顯著性、R²、f²、Q²predict
     │
     ▼
SQLite 資料庫
  - 研究宣告（含時間戳記）
  - 上傳的每一版資料
  - 每一次分析/優化操作的完整、不可竄改紀錄
```

### 2.2 設計原則：三篇方法論文獻決定了架構的形狀

這套系統的架構不是憑空設計的，三個核心設計決策各自對應一篇方法論文獻：

**原則一：測量模型與結構模型必須分兩階段評估，且前者是後者的硬性前提**

> Hair, J. F., Hult, G. T. M., Ringle, C. M., & Sarstedt, M. *A Primer on Partial Least Squares Structural Equation Modeling (PLS-SEM)*. Sage Publications.

PLS-SEM 的標準流程是先確認測量模型的信效度過關，才進行結構模型的路徑顯著性檢定。系統把這件事做成程式碼層面的**強制關卡**：如果測量模型（信度、AVE）沒有通過門檻，系統會直接拒絕執行結構模型分析（回傳 403 錯誤），除非使用者明確登記理由強制略過，而這個略過動作會被完整記錄下來。

**原則二：排除任何一筆樣本都需要「統計 + 實質」雙重理由，不能只憑一個統計數字**

> Curran, P. G. (2016). Methods for the detection of carelessly invalid responses in survey data.

單一指標（例如只看 Cook's Distance 這類統計影響力指標）並不足以構成「這筆資料有問題該排除」的理由。系統實作了多訊號收斂偵測（Mahalanobis 距離、組內作答變異量 IRV、連續重複作答 Long-string），**必須至少兩個獨立訊號同時成立**，才會把某筆樣本標記為「建議複查」；而系統的自動優化引擎在考慮要不要排除某筆樣本以換取路徑顯著性時，**同時要求這筆樣本統計上有影響力、又符合實質理由標記，兩個條件缺一不可**才會執行排除。

**原則三：驗證性與探索性分析必須明確標示，否則等同於未揭露的研究者自由度**

> Simmons, J. P., Nelson, L. D., & Simonsohn, U. (2011). False-Positive Psychology: Undisclosed Flexibility in Data Collection and Analysis Allows Presenting Anything as Significant. *Psychological Science*.

Simmons 等人證明，未揭露的分析彈性（例如可以自由決定要不要多刪幾筆資料）可以把名目 5% 的偽陽性率推高到 60%。系統要求研究者在跑任何分析之前，先透過「宣告」機制把理論假設（構面、題項、假設路徑）記錄下來並蓋上時間戳記，作為驗證性分析的基準點；任何在此之後執行的「搜尋顯著性」動作，都在系統的操作紀錄裡被明確標記為探索性（exploratory），跟原始宣告的驗證性假設清楚分開。

### 2.3 系統分層架構（L0 至 L6）

| 層級 | 名稱 | 功能 |
|---|---|---|
| L0 | 研究設計宣告層 | 上傳資料前，先宣告構面、題項、假設路徑，作為驗證性/探索性分析的分界時間點 |
| L1 | 資料品質層 | 多訊號收斂偵測可能隨意作答的樣本，只診斷不刪除 |
| L2 | 測量模型層 | 信度、收斂效度、區辨效度檢定，是否過關決定能不能繼續往下分析 |
| L3 | 結構模型層 | 路徑顯著性、解釋力、預測力等結構模型指標 |
| L4 | 自動優化引擎 | 分兩階段自動搜尋：先確保測量模型過關，再針對不顯著的路徑嘗試找出最小幅度的合理調整 |
| L5 | 審計與版本層 | 每一步操作的完整、不可竄改紀錄，可完整重放從原始資料到最終數字的每一步 |
| L6 | 呈現層 | 使用者操作介面 |

### 2.4 統計計算細節

**測量模型（L2）指標與門檻依據：**

| 指標 | 意義 | 門檻 | 依據 |
|---|---|---|---|
| Cronbach's α | 內部一致性信度 | ≥ 0.7 | 學界通用門檻 |
| CR（組合信度） | 修正 α 的信度指標 | ≥ 0.7 | 同上 |
| AVE（平均變異萃取量） | 收斂效度 | ≥ 0.5 | 同上 |
| HTMT | 區辨效度，比傳統 Fornell-Larcker 準則更敏感 | 嚴格 0.85 / 寬鬆 0.90 | Henseler, Ringle & Sarstedt (2015) |
| 多因子 EFA + Parallel Analysis | 驗證題項分組（構面）本身是否合理，而非只驗證分組內部的信效度 | 因子保留數由 Parallel Analysis 決定，非人工主觀判斷 | 心理計量學標準方法 |

**結構模型（L3）指標：**

| 指標 | 意義 |
|---|---|
| 路徑係數 + Bootstrap 顯著性 | 假設路徑是否成立 |
| R² | 模型解釋力 |
| f² | 個別前因變數對解釋力的效果量貢獻 |
| VIF | 共線性診斷 |
| Q²predict（PLSpredict） | 樣本外預測力，採用 Shmueli et al. (2019) 提出的現代做法，取代傳統但爭議較大的 blindfolding 方法 |

需特別說明：**SRMR（標準化殘差均方根，整體配適度指標）目前刻意未實作**。PLS-SEM 並不像傳統共變異數導向 SEM（CB-SEM）有一套幾乎強制報告配適度指標的傳統，SRMR 是後來才被借用過來的近似指標，其門檻在 PLS-SEM 情境下並未被嚴謹驗證，連 Hair 等人教科書都提醒使用時需謹慎。經評估，本系統已具備的信效度與解釋/預測力指標已涵蓋 PLS-SEM 論文的必要報告項目，SRMR 非必要項目，貿然自行實作一個沒有驗證管道的公式，風險大於效益。

### 2.5 自動優化引擎：分兩階段搜尋演算法

這是系統的核心功能，用來處理「測量模型或結構模型未達標時，該如何以最小幅度的合理調整讓分析結果站得住腳」這個問題，取代傳統上研究者手動反覆嘗試刪題項/刪樣本的過程。

**Stage A（測量模型關卡）**：對每個未通過信效度標準的構面，採用貪婪演算法，優先刪除因素負荷量最低的題項，直到 AVE 達標或剩兩題仍未達標為止（後者會回報「無法自動修復，建議研究者重新檢視此構面」，不會勉強繼續）。這個階段是**強制關卡**——沒有全數通過，後續的結構模型搜尋完全不會執行。

**Stage B（結構顯著性搜尋，僅在 Stage A 全部通過後才啟動）**：針對每一條尚未達到統計顯著的假設路徑，各自獨立搜尋。搜尋方式是計算每筆樣本對該路徑迴歸結果的統計影響力（Cook's Distance），依影響力大小排序，逐步嘗試排除；**但每一筆候選排除的樣本，都必須同時是 L1 資料品質層標記過的「疑似隨意作答」樣本，兩個條件缺一不可才會真的排除**，且刪除比例設有上限（預設 10%）。若在此限制內仍找不到讓路徑顯著的排除組合，系統會如實回報「無法在有實質理由的前提下找到解法」，並附上給研究者參考的檢討建議（例如：是否有中介變數、是否需要重新檢視理論架構），**但不會自動刪除構面或改變理論模型**。

這個雙重限制（統計影響力 + 實質理由）是整套系統方法論嚴謹度的核心：實測驗證過，同一組資料，如果拿掉「必須有實質理由」這個限制，統計搜尋確實能找到讓某條沒有真實效果的路徑「看起來顯著」的刪除方式；加回這個限制後，系統正確地拒絕了這個搜尋，只留下建議研究者人工判斷——避免系統淪為自動化的資料操弄（p-hacking）工具。

### 2.6 部署環境

系統以 Docker Compose 管理三個容器：Python/FastAPI 主服務（內含 R 執行環境）、Gradio 對話式前端、Streamlit 分步驟前端，兩個前端各自獨立部署（不同 port），共用同一個後端與 session 機制。統計資料庫採用 SQLite，優先考量是部署簡便、零額外設定成本，符合目前開發階段的規模需求，未來若有正式上線的量能需求，可平順遷移至 PostgreSQL 等正式資料庫，不需重寫應用邏輯。

## 三、目前缺口與後續規劃

| 項目 | 現況 | 說明 |
|---|---|---|
| SRMR | 刻意不做 | 見 2.4 說明，非必要指標，暫不列入後續計畫 |
| 情境並列比較 | ✅ 已完成 | 每次 `/optimize/full-search` 都可選填標籤，事後可從審計歷程多選 2-4 次搜尋並排比較 |
| 多輪對話介面 | ✅ 已完成 | 新增 Gradio 單一對話式介面，支援上傳、宣告、跑診斷、討論、依使用者要求調整參數重新分析，全程透過 LLM tool calling 呼叫既有分析函式，不繞過任何既有關卡；原本的 Streamlit 分步驟頁面保留並存，兩者共用同一套後端 |
| 審計履歷匯出成論文附錄格式 | 未實作 | 目前只能透過 API／對話介面查詢，尚無一鍵匯出成可附進論文的格式 |
| 問卷排版與樣本代表性 | 明確排除在系統範疇外 | 這屬於問卷設計與發放階段的工作，跟本系統「分析已回收資料」的定位不同，經評估不適合勉強塞入同一套系統 |

**優先順序建議**：剩餘缺口只有審計履歷匯出，可視時間安排；情境並列比較與多輪對話介面已在後續階段完成，詳見 `DEVELOPMENT_LOG.md` 階段 12、14。

**這套系統的學術定位補充說明**：經文獻檢索，目前 PLS-SEM 分析工具（SmartPLS、JASP 及各類 R 套件）多半止步於「算出指標、標示是否達標」，是否調整、如何調整仍需研究者自行手動判斷；近期雖有學術文獻提出用基因演算法等方法自動搜尋結構模型設定（如 Automated specification search for composite-based structural equation modeling: A genetic approach），但聚焦於結構模型的全域搜尋，跟本系統鎖定測量模型信效度修復、並以可解釋的啟發式演算法（而非黑箱式全域搜尋）搭配實質理由把關的設計方向不同。這代表本系統目前實作的「自動化診斷+優化建議」功能，在文獻中屬於相對少被涵蓋的區塊，或可作為論文貢獻點之一。
